A Simple File System

The goal of this project is to design and implement a simple file system called cpmFS (i.e., CP/M file sytem). Through the late 1970s and into the mid-1980s, CP/M (Control Program for Microcomputers) - a disk-based operating system - had dominated its era as much as MS-DOS and later Windows dominated the IBM PC world. CP/M is clearly not the last word in advanced file systems, but it is simple, fast, and can be implemented by a competent programmer in less than a week.

Our simple file system will allow users to list directory entries, rename files, copy files, delete files, as well as read/write/open/close files. We will use a version of the CP/M file system used on 5.25” and 8” floppy disks in the 1970's (support for CP/M file systems is still included in Linux to this day). We will develop our code in C. This can be done on any computer with an ANSI C compiler (e,g. gcc, clang, etc.). We will not be modifying the linux kernel but developing a stand-alone program (i.e., a simulated filesystem).

Extent

Every file within our file system will have an associated extent. The extenet (also called the File Control Block or Directory Entry) is a struct that contains exactly 32 bytes. These 32 bytes are split into two 16-byte sections. The first 16-byte section contains basic information about the file. The first byte of this section contains the file status. To delete a file, we simple mark its status as invalid. The next next 8 bytes contain the file name. The next 3 bytes are reserved for the file extension name. The final 4 bytes are reserved for XL,BC,XH, and RC. We will cover these later but for now we will say that XL, and XH are reserved/unused for if we want to change the file system structure in the future. For now, we will not be using those two bytes.

The last 16-byte section contains block pointers (also referred to as block addresses). Each block pointer is a 8 bits (1 byte) that points to a single block address space. Because the second section contains 16-bytes, and each pointer is a single byte, each file can contain up to 16 blocks. Each block contains 1024 bytes (1kB). Our FS can support in total 2^8= 256 blocks. Thus, the address stored by our block pointer can be anywhere from 0-255. Block 0 contains all extents in our FS. We refer to this block as a Super Block. Because this block (like all others) is 1024 bytes in size, it can hold a maximum of

1024 \text{ Bytes}/ \frac{32 \text{ Bytes}}{\text{ extent}} = 32 \text{ extents}

This means our FS can contain up to 32 files. However, If we assume each file uses exactly 16 bytes, we can have up to

256 \text{ Blocks}/\frac{32 \text{ Bytes}}{\text{extenet}} = 16 \text{ Files}

Disk

Our simulated hard disk will be stored in a file named image1.img. This disk will contain 256 blocks as previously stated. Block 0 will be reserved for the file system itself. It will contain the extents for every file as previously stated. Each block contains 1-8 sectors. Each sector can have up to 128 bytes. Every block other than the last block is “fully loaded”. The last block is “partial filled” meaning some sectors are not filled. We say the number of filled sectors is the RC value. The amount of space used in the partially filled sector in the last block is the BC value.

File Size

To calculate the file size, we will use the following equation

fileSize = NB \cdot 1024 \text{ Bytes} + RC \cdot 128 \text{ Bytes} + BC

Where NB is the number of fully loaded blocks, RC is the number of sectors in last block, and BC is the number of bytes in the last sector. To get NB we will iterate over the blocks in the file extent until we find a null pointer indicating that block is not assigned. RC and BC are given in the 4-byte section after the file extension.

Reading Disks

The disk variable is a globally allocated memory space which stores the bytes contained within the disk.

The readImage() function takes in a file name string and reads the contents of the file into the disk variable.

Reading Blocks

The disk is comprised of 256 blocks. To access the data of a single block, we will use a function called blockRead(). This function takes a pointer to a buffer and a blockNum value, and reads the contents of the specified block from the disk array into the buffer.

Reading Extents

Extents are stored in block 0 of the disk. Block 0 contains a total of 32 extents. To read a single extent from the disk, we will be defining our own function mkDirStruct() within the cpmfsys.c file of the source code. This function will load the data stored within the extent into a dirStructType data structure. We will refer to block 0 by defining a pointer we will name "e". To extract a single extent, we will use another pointer named "d".

From this point, our goal will be developing the following functions in source code file cpmfsys.c:

1. mkDirStruct()

Given a pointer to block 0 (e) and an extent index, this function allocates memory and populates a DirStructType

2. writeDirStruct()

writes contents of a DirStructType to the given extent index in block 0 (e)

3. makeFreeList()

generates a boolean list (FreeList) of all blocks that are free/unused. The data structure for the free list is a bit vector.

4. printFreeList()

debugging function which prints a 16x16 array of the free blocks given by the FreeList. Each row is prefixed by the hex address of the first block in that row. We will use a * symbol to denote a used block

5. cpmDir()

Prints the file directory to stdout. Each filename is printed on its own line, with the file size, in base 10.

6. checkLegalName()

Internal function that checks if file name is valid (name or extension too long, name blank, or illegal characters in name or extension). Returns true for legal name, and false for illegal.

7. findExtentWithName()

Internal function which returns the extent nunber 0-31. If name is illegal or not found, returns -1.

8. cpmDelete()

deletes the file named "name", and frees its disk blocks in the free list.

9. cpmRename()

reads directory block, modifies the extent for file named oldName with newName, and writes to the disk.
